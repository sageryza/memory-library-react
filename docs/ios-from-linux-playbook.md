# Shipping a SwiftUI app to TestFlight from a Mac‑less / CI environment — playbook

How we took a brand-new SwiftUI app from *nothing* to **installable on a phone via
TestFlight**, built entirely on a cloud macOS runner with **no local Mac**. This is
the battle-tested version: every gotcha we hit and the exact fix. Use it to repeat
the process for another app (e.g. XI).

---

## 0. The shape of it

```
push code → GitHub Actions macOS runner → XcodeGen makes the .xcodeproj
          → xcodebuild archive + export (signed via App Store Connect API key)
          → fastlane upload_to_testflight → App Store Connect → TestFlight → phone
```

- **No Mac needed.** GitHub's hosted **macOS runners** are real Macs; they do the build.
- **No certificates to manage by hand.** An App Store Connect **API key** + Xcode's
  `-allowProvisioningUpdates` lets Apple create the distribution cert + profile on the fly.
- Project is generated by **XcodeGen** from `project.yml` (so no `.xcodeproj` in git).
- **fastlane** only for the upload step; archive/export are raw `xcodebuild` (clearer errors).

---

## 1. Accounts / one-time prerequisites

- **Apple Developer Program** ($99/yr) — required for TestFlight + signing.
- **App Store Connect**: an **app record** must exist for the bundle id before upload.
- A **Firebase project** if the app uses Firebase (this one does: anon auth + Functions).

---

## 2. The keys/secrets you actually need

Store these as **GitHub repo secrets** (Settings → Secrets and variables → Actions).
Names below match the workflow.

| Secret | What it is | How to get it | Notes |
|---|---|---|---|
| `ASC_KEY_ID` | App Store Connect API key id (~10 chars) | App Store Connect → Users and Access → Integrations → App Store Connect API → **Generate** | **Role must be `Admin`** (see gotcha #7). Account-wide → reuse across apps. |
| `ASC_ISSUER_ID` | Account issuer id (UUID) | Same page, shown at top | Not secret; account-wide. |
| `ASC_KEY_P8` | The `.p8` private key, **base64-encoded** | Download the `.p8` once, then `base64 -i AuthKey_XXXX.p8 \| pbcopy` | Store base64, not raw (gotcha #8). Account-wide. |
| `APPLE_TEAM_ID` | 10-char team id | developer.apple.com/account → Membership | Account-wide. Can hardcode in workflow. |
| Firebase config | `GoogleService-Info.plist` (per app) | Firebase console → Add app → iOS, or via the Management API (see §5) | Client config, safe to commit in a private repo, or inject from a base64 secret. |

> **Account-wide vs per-app:** `ASC_*`, `APPLE_TEAM_ID` are the *same* for every app in
> your account — **reuse them**, don't regenerate. Per-app: bundle id, the Firebase iOS
> app + plist, the App Store Connect app record, the app icon.

**A Firebase *Admin SDK service account* JSON is NOT any of these.** It's a server master
key — never ship it in an app. We used one *transiently* to call the Firebase Management
API (§5), then revoked it.

## 4. The gotchas — every one we hit, and the fix

1. **`xcodebuild: project is in a future Xcode project file format (77)`**
   XcodeGen ≥2.45 emits the Xcode 16 format; the runner defaulted to Xcode 15.4.
   → **Select a newer Xcode** in the workflow (`sudo xcode-select -s /Applications/Xcode_26*.app`).

2. **`xcodebuild -showBuildSettings timed out after 4 retries (base 3s)`**
   Firebase pulls ~14 SPM packages; resolving build settings is slow.
   → Env `FASTLANE_XCODEBUILD_SETTINGS_TIMEOUT: "180"` (+ `_RETRIES: "6"`).

3. **fastlane `produce` to auto-create the App Store Connect app record failed**
   (`No value found for 'username'` / `api_key` not an option).
   → Easiest: **create the app record manually** in App Store Connect (or via API). Don't
   fight `produce`.

4. **`build_app`/gym doesn't accept `api_key`** (in this fastlane version).
   → Pass auth straight to xcodebuild via xcargs:
   `-authenticationKeyID <id> -authenticationKeyIssuerID <iss> -authenticationKeyPath <p8>`.

5. **Export step failed: `Looks like no provisioning profile mapping was provided`**
   gym's export wrapper insists on a manual profile map even for automatic signing.
   → **Drive `xcodebuild -exportArchive` directly** (don't use gym's export). Bonus: real
   errors print to the log instead of being swallowed.

6. **`xcodebuild -exportArchive` also needs the auth flags** (separate process from archive).
   → Pass the same `-authenticationKey*` + `-allowProvisioningUpdates` to the export call too.

7. **`exportArchive Cloud signing permission error` / `No profiles for '<bundle>' were found`**
   The App Store Connect API key was **App Manager** role. Cloud-managed *distribution*
   signing requires **Admin**. App Manager can archive but cannot create the distribution cert.
   → **Regenerate the API key with `Admin` access.** (Key roles are fixed at creation.)

8. **`The data couldn't be read because it isn't in the correct format`** (during signing)
   The `.p8` got corrupted pasting a multi-line key into a secret.
   → **Store the key base64-encoded** and decode it in the Fastfile
   (`Base64.decode64(ENV["ASC_KEY_P8"])`, and `is_key_content_base64: true`).

9. **Apple upload validation: `Missing CFBundleIconName` / `no 120×120 icon`**
   → Add an **AppIcon asset catalog** with a single 1024×1024 PNG (Xcode generates the rest),
   and set `ASSETCATALOG_COMPILER_APPICON_NAME: AppIcon`.

10. **Apple upload validation: `built with iOS 18.5 SDK; iOS 26 SDK required`**
    Apple enforces a current-SDK floor. → **Build with Xcode 26** (see gotcha #1; we made the
    workflow fail fast if the runner's newest Xcode is < 26).

11. **Apple upload validation: `No orientations were specified`**
    → Set `INFOPLIST_KEY_UISupportedInterfaceOrientations_iPhone` / `_iPad`.

12. **App installs but crashes instantly: `+[FIRApp configure]` raises (SIGABRT)**
    No `GoogleService-Info.plist` in the bundle.
    → Bundle the plist (commit it, or inject from a secret before XcodeGen runs). We also
    added an explicit-options fallback in `FirebaseApp.configure` so a missing file can't crash it.

13. **Phone keeps showing the OLD build / TestFlight asks for an "invite code"**
    Two things, both **not** code problems:
    - **Export compliance** must be answered for each build (`Missing Compliance`). Until then
      the build is invisible to testers. Set it via API: `PATCH /v1/builds/{id}` with
      `usesNonExemptEncryption: false` (or add `ITSAppUsesNonExemptEncryption=NO` to Info.plist).
    - **Internal testers** auto-receive every *compliant* build — you don't add builds to an
      internal group. Just add yourself as a tester once. ("Invite code" simply means nothing
      is assigned to your Apple ID yet.)

## 5. Automation superpowers (do these from any Linux box / chat)

You can drive most of Apple + Firebase over HTTPS with the keys above. We used these to avoid
console clicking. Sign JWTs with `openssl` (env `cryptography` libs are often broken).

**Firebase Management API** — create the iOS app + fetch its config, no console:
```
# auth: service-account JWT (RS256) → token; scope cloud-platform
POST https://firebase.googleapis.com/v1beta1/projects/{PROJECT}/iosApps  {bundleId, displayName}
GET  .../iosApps/{appId}/config        # returns base64 GoogleService-Info.plist
```

**App Store Connect API** — ES256 JWT signed with the `.p8` (`kid`=KEY_ID, `iss`=ISSUER,
`aud`=appstoreconnect-v1). Endpoints we used:
- `GET  /v1/apps?filter[bundleId]=...` — find the app id
- `GET  /v1/builds?filter[app]=...` — build list + `processingState` + `usesNonExemptEncryption`
- `PATCH /v1/builds/{id}` `{usesNonExemptEncryption:false}` — clear "Missing Compliance"
- `GET  /v1/apps/{id}/betaFeedbackCrashSubmissions` → `.../crashLog` — **read symbolicated
  crash logs directly** (so you don't need the tester to screenshot)
- `GET  /v1/betaGroups?filter[app]=...` — tester groups

> ES256 detail: `openssl dgst -sha256 -sign` emits a **DER** signature; JWT needs the raw
> r‖s (two 32-byte ints) — parse the DER and concatenate, then base64url.

## 6. Reusing this for another app (XI)

Make it a **separate iOS app** (own bundle id, own Firebase iOS app + plist, own App Store
Connect record, own icon). **Reuse** the pipeline pattern and the **account-wide** keys
(`ASC_*`, `APPLE_TEAM_ID`). Concretely, for XI you'd:

1. Copy `ios/`, `ios/fastlane/`, `.github/workflows/ios-testflight.yml`; change bundle id,
   app name, scheme, icon.
2. Register `com.sageryza.xi` (or whatever) as a Firebase iOS app → get its plist (§5 or console).
3. Create the App Store Connect app record for that bundle id.
4. Add the **same** `ASC_*` + `APPLE_TEAM_ID` secrets to that repo. Push. Done.

The only genuinely new per-app work is the icon, the bundle id, and the per-app Firebase plist.
